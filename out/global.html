CommentingCode

Be generous with comments. It is useful to leave information that will be read at a later time by people (possibly yourself) who will need to understand what you have done. The comments should be well-written and clear, just like the code they are annotating. An occasional nugget of humor is appreciated, but cynical and resentful comments are not.

You do not need to write comments that describe what is easy to see in the code. Since your code should be clear, concise and well-written, additional comments describing what the code is doing are not necessary. Instead, leave comments to provide additional information and focus on the WHY. For example, WHY is the code doing what it is doing? WHY was a certain choice made? Or WHERE can additional information be found?

If you use a workaround or hack, denote that with a “ //HACK: ” and then explain why that code was hacked or a workaround was used.

If something is not complete, or has to be changed down the road, denote that with “ //TODO: “ and then explain what is still needed.

Make comments meaningful. Focus on what is not immediately obvious.

It is important that comments be kept up-to-date. Erroneous comments can make programs even harder to read and understand.

NamingConventions

Please follow the conventions defined below when naming classes, variables, fields and methods in your code:

For naming variables and fields, use camel-case notation, such as “variableNamesLikeThis”.

For naming classes and enums, use Pascal-Case notation, such as “ClassNamesLikeThis” and “EnumNamesLikeThis”.

For constants, use all-caps separated by underscores, such as “CONSTANT_VALUES_LIKE_THIS”.

For namespaces, use camel-case notation, such as “foo.namespaceNamesLikeThis.bar”.

In general, use functionNamesLikeThis, variableNamesLikeThis, ClassNamesLikeThis, EnumNamesLikeThis, methodNamesLikeThis, CONSTANT_VALUES_LIKE_THIS, foo.namespaceNamesLikeThis.bar, and filenameslikethis.js.

Names should be formed from the 26 upper and lower case letters (A .. Z, a .. z), the 10 digits (0 .. 9), and _ (underscore). Avoid use of international characters because they may not read well or be understood everywhere.

Most variables and functions should start with a lower case letter.

SemanticMarkUp

With the increased emphasis on semantics and structure in HTML 5 it is important to consider the following when marking up your web and or web application pages:

  • Metadata: Content that sets up presentation or behavior of the content
  • Embedding: Any content that imports or resources into the document. Those elements are as follows: audio, canvas, embed, iframe, img, math, object, svg, and video
  • Interactive: Any content specifically intended for interaction. Those elements are as follows: a, audio, button, details, embed, iframe, img, input, keygen, label, menu, object, select, textarea and video
  • Heading: Defines the header of a section which can either be explicitly marked up with sectioning elements or implied by the heading content itself. Those elements are as followed: h1-h6 and hgroup.
  • Phrasing: The text of the document, as well as elements used to markup the text within the paragraph level structures. Those elements are as followed: a, abbr, audio, b, bdo, br, button, canvas, cite, code, command, datalist, del, dfn, em, embed, i, iframe, img, input, ins, kbd, keygen, label, link, map, mark, math, meta, meter, noscript, object, output, progress, q, ruby, amp, script, select, small,, span, strong, sub, sup, svg, textarea, time, var, video and wbr.
  • Flow: Majority of elements that would be included in the normal flow of a document. Those elements are as followed: a, abbr, address, article, aside, audio, b, blockquote, bdo, br, button, canvas, cite, code, command, datalist, details, div, dl, del, dfn, em, embed, fieldset, figure, footer, form, h1, h2, h3, h4, h5, h6, header, hgroup, hr, i, iframe, img, input, ins, kbd, keygen, label, link, map, mark, math, meta, meter, noscript, object, output, p, pre, progress, q, ruby, amp, section, script, select, small,, span, strong, sub, sup, svg, table, textarea, time, var, video and wbr.
  • Sectioning: Content that defines the scope of headings and footers. Those elements are as followed: article, aside, nav and section
  • HTML 5 is concerned with the structure of the document not the presentation. The display of elements is left entirely to the user agent and the CSS employed.
  • What’s the role of div tags? Previously div tags where widely used to mark up semantics into common to a group of consecutive elements. With all the new tags for structuring content in HTML 5, div tags should be used when the HTML 5 tags are not appropriate. Div tags are still the best choice for adding structure. Always make sure semantic tags are appropriate.
  • Only use section and article tags when sectioning content is appropriate. Every time you use a section or article tag you create a new title element in your outline. If the section you are creating doesn't warrant this level of specificity consider using a div.
  • Only use the aside tag for related content
  • Headers and footers are for grouping heading or footer content
  • Nav tags should not be used for every navigation grouping
  • Divs can used to group related content
  • Outlining

StyleSheetAuthoringKeepItSassy

Stylesheet authoring at a large scale can become easily difficult to manage, the slight variations of properties can easily get a bit out of hand. Using Sass affords our teams several features to combats CSS's shortcomings. Such as

  • Extends
  • Functions
  • Mixins
  • Loops
  • List
  • Nesting
  • Variables
  • Importing
  • Math
  • Comparisons
  • Interpulation [Shimming]
  • SassScript

These are just a few features that make authoring stylesheets easier to maintain and a bit more fun. There are a few things to keep in mind when getting Sassy.

  • As of this documentation please use the Sassy CSS file extension (.scss)
  • Use the Sassy CSS(.scss) syntax
  • Nest Sparingly, if you find yourself nesting too deep consider refactoring
  • Importing happens on compile rather client side, this provides a way for us to abstract modules into partials
  • Sprite mindfully, spriting is perfect for simple animations, UI assets etc...
  • Sprites beyond 8192 pixels wide or tall will not render on mobile devices and are performance issues. (Believe it or not we have third parties deliver us sprites larger than that)
  • Keep it DRY
  • Extend
  • Include mix-ins
  • Utilize $variables for easy site wide updates
  • Use Sass Placeholders for reusable styles that don't warrant a mix-in
  • Keep it Sassy
  • Compress and minify production stylesheets

StyleSheetAuthoringNamingConventions

In general, please use fully-structural naming conventions over presentational naming conventions. Below are some additional guidelines to assist you while developing web applications and sites.

  • Use what's best for the project at hand.
  • When assigning a class or id, ask yourself “What is this element for?
  • What role does this piece of the document play
  • Consider the document structure
  • Avoid using names that rely on locational or visual aspects of the particular element
  • Use names that are intuitive to you
  • Think Modular
  • Think Semantically
  • Names should be formed from the 26 upper and lower case letters (A .. Z, a .. z), the 10 digits (0 .. 9), and - (hyphen). Avoid use of international characters because they may not read well or be understood everywhere.
  • Most variables, mix-ins and functions should start with a lower case letter.
  • All CSS selectors (classes or IDs) are lowercase, multiple words are separated with a hyphen, no underscore nor camel-case.
  • All CSS definitions should take place in a CSS file or in a CSS part at the top of the page, inline styles are highly discouraged.
  • It is encouraged to use generic classes for common styling issues instead of "id".
  • Avoid conflicts in naming scheme

Variables

The most important thing you can do is to choose an identifier name that is both meaningful and correct. For instance, if your program deals with customer accounts then your naming choices should reflect that (e.g., customerName, accountDetails).

When you fail to specify var, the variable gets placed in the global context, potentially clobbering existing values. Also, if there's no declaration, it's hard to tell in what scope a variable lives (e.g., it could be in the Document or Window just as easily as in the local scope). So always declare with var.

Use of global variables should be minimized.

Use camelCase when defining variable names

Use all caps followed by underscores when defining constant variable names

Primitive types (number, string, boolean) are constant values.

Omit var in long lists of variables

Cache your jQuery objects: If any variable and or object you create is utilizing jQuery to create those objects make sure you use the $ sign in your variable naming convention
